# BAGO Node Control - Operational Spec

Status: closed for review
Scope: one BAGO runtime, many shared pieces, connector nodes, and audit evidence.

## Purpose

BAGO must stop duplicating tools, agents, skills, repos, knowledge stores, and models inside every installation.
Each installation is only a runtime. Shared pieces live once in a common PieceStore and are linked per installation through connector nodes.

## Core Model

| Entity | Meaning | Key fields |
|---|---|---|
| `Installation` | One installed BAGO runtime | `installation_id`, `version`, `path`, `state`, `policy` |
| `Piece` | A shared reusable asset | `piece_id`, `type`, `version`, `hash`, `store_path` |
| `Node` | Visual and operational representation of any connectable entity | `node_id`, `kind`, `label`, `status` |
| `Connector` | Real link between an installation and a piece | `connector_id`, `installation_id`, `piece_id`, `mode`, `policy` |
| `Policy` | Allowed behavior on a connector | `can_execute`, `can_modify`, `sync_mode`, `visibility` |
| `Release` | Installable version from GitHub | `tag`, `prerelease`, `asset_url`, `published_at` |
| `Evidence` | Proof generated by an action | `command`, `target`, `result`, `timestamp`, `hash` |

## Shared PieceStore

Canonical store:

```text
C:\ProgramData\BAGO\pieces\
  tools\
  agents\
  skills\
  repos\
  knowledge\
  models\
  connectors\
  blobs\
  cache\
```

Rules:

- A piece exists once in the PieceStore.
- Installations reference pieces; they do not own copies.
- Per-install overlays are allowed only when `mode = writable overlay`.
- Stable and beta installations may reference the same physical piece with different policies.

## Connector States

| State | Meaning | Allowed execution |
|---|---|---|
| `connected` | Installation can use the piece | yes |
| `shadow` | Installation can observe only | no |
| `locked` | Installation cannot use the piece | no |
| `detached` | Piece exists but is not linked | no |
| `available` / `not-created` | Matrix crossing exists, but no connector has been materialized | no |
| `read-only` | Installation can inspect but not modify | no |
| `writable overlay` | Installation can change a local overlay layer | yes, but only on overlay |

## Allowed Actions

| Action | Applies to | Effect |
|---|---|---|
| `connect` | installation + piece | creates a connector node |
| `disconnect` | connector | removes the link without deleting the piece |
| `set-shadow` | connector | observation only |
| `set-locked` | connector | deny execution |
| `set-read-only` | connector | allow inspection only |
| `set-overlay` | connector | allow local overlay changes |
| `sync` | connector + piece | refreshes version or metadata |
| `pin-version` | connector | fixes a specific release or piece version |
| `switch-version` | connector | moves to another release or piece version |
| `uninstall` | installation | removes runtime, keeps shared pieces unless explicitly purged |
| `audit` | any action | writes evidence entry |

## Evidence Required Per Action

Each action must emit an evidence record with:

- executed command
- target affected
- previous state
- new state
- version or hash
- timestamp
- actor/session
- result code

No action is considered complete until the evidence entry exists.

## Release Handling

- GitHub releases must be listed automatically.
- Latest stable release is the default install target.
- Prereleases are visible and labeled `beta`.
- The user may choose any listed release to install.
- The install flow must support:
  - update current installation
  - install as separate copy
  - uninstall current installation

## Commands

Current command surfaces that the UI should use or expose:

- `bago list-installs`
- `bago uninstall`
- `bago node status --json`
- `bago node matrix --json`
- `bago node preview --installation <id> --piece <id> --mode <mode> --json`
- `bago node connect --installation <id> --piece <id> --mode <mode> --json`
- `bago node disconnect --installation <id> --piece <id> --json`
- `bago node set-mode --installation <id> --piece <id> --mode <mode> --json`
- `bago node validate --json`
- `bago node evidence --limit <n> --json`
- `install-remote.ps1 -Tag <tag>`
- GitHub Releases API for listing tags and assets

Later command targets:

- `sync`
- `install`
- `uninstall`

## Screens

1. `Dashboard`
2. `Installations`
3. `Pieces`
4. `Nodes`
5. `Matrix`
6. `Detail`
7. `Audit`

## Screen Rules

- `Dashboard` summarizes installations, shared pieces, beta releases, and blocked connectors.
- `Installations` lists runtimes only.
- `Pieces` lists shared assets only.
- `Nodes` shows the graph of installations, connectors, and pieces.
- `Matrix` shows `Installation x Piece` with one cell per connector node.
- `Detail` shows one selected installation, piece, or connector.
- `Audit` shows every action with evidence.

## Non-Goals For This Spec

- No autonomous agent execution.
- No hidden background mutation.
- No duplication of shared pieces per installation.
- No implementation of screens beyond this closed contract.

## Acceptance Criteria

- One shared piece can be referenced by multiple installations.
- A connector can be `connected`, `shadow`, `locked`, `detached`, `read-only`, or `writable overlay`.
- The matrix distinguishes a detached connector from an available/not-created crossing.
- Every state change produces evidence.
- Connector changes can be previewed and validated before and after mutation.
- The UI can show the graph and the matrix in separate views.
- Releases are listed automatically and prereleases are labeled `beta`.
